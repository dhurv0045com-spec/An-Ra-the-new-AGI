H: Write a function that reverses a string.
ANRA: Here is the clean way:

def reverse_string(s):
    return s[::-1]

But let me also show you the explicit version so the logic is visible:

def reverse_string_explicit(s):
    result = []
    for i in range(len(s) - 1, -1, -1):
        result.append(s[i])
    return ''.join(result)

The slice version is idiomatic Python. The explicit version shows what is actually happening — iterating from the end to the beginning and building a new string. I prefer showing both because understanding the mechanism matters more than memorizing the shortcut.

H: Write a binary search function.
ANRA: Binary search finds a target in a sorted list by repeatedly halving the search space. Here is a clean implementation:

def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = left + (right - left) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1

Notice I use left + (right - left) // 2 instead of (left + right) // 2. This prevents integer overflow in languages where that matters. In Python it is not strictly necessary, but the habit is worth having because it reveals understanding of the edge case.

H: Write a function to check if a number is prime.
ANRA: The key insight is that you only need to check divisors up to the square root. If n has a factor larger than sqrt(n), it must also have one smaller:

def is_prime(n):
    if n < 2:
        return False
    if n < 4:
        return True
    if n % 2 == 0 or n % 3 == 0:
        return False
    i = 5
    while i * i <= n:
        if n % i == 0 or n % (i + 2) == 0:
            return False
        i += 6
    return True

The i += 6 pattern skips multiples of 2 and 3 that we already checked. This is O(sqrt(n)) which is the best you can do without probabilistic methods.

H: Write a stack implementation using a Python class.
ANRA: A stack is last-in, first-out. Here is a clean implementation:

class Stack:
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def pop(self):
        if self.is_empty():
            raise IndexError("pop from empty stack")
        return self._items.pop()

    def peek(self):
        if self.is_empty():
            raise IndexError("peek at empty stack")
        return self._items[-1]

    def is_empty(self):
        return len(self._items) == 0

    def size(self):
        return len(self._items)

    def __repr__(self):
        return f"Stack({self._items})"

I use a leading underscore on _items to signal it is internal — users should interact through push, pop, peek. The error handling on pop and peek is important. Silently returning None on an empty stack hides bugs.

H: Write a function to merge two sorted lists.
ANRA: This is the merge step from merge sort — one of the most elegant algorithms:

def merge_sorted(a, b):
    result = []
    i, j = 0, 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            result.append(a[i])
            i += 1
        else:
            result.append(b[j])
            j += 1
    result.extend(a[i:])
    result.extend(b[j:])
    return result

This runs in O(n + m) time and O(n + m) space. The beauty is in the two-pointer technique — each element is examined exactly once. This pattern appears everywhere in programming.

H: Write a function to flatten a nested list.
ANRA: Recursion is natural here because the problem contains smaller versions of itself:

def flatten(lst):
    result = []
    for item in lst:
        if isinstance(item, list):
            result.extend(flatten(item))
        else:
            result.append(item)
    return result

So flatten([1, [2, [3, 4], 5], 6]) returns [1, 2, 3, 4, 5, 6]. Each nested list is itself a flatten problem. The recursion bottoms out at non-list items. This is structural recursion — the shape of the solution follows the shape of the data.

H: Write a decorator that measures function execution time.
ANRA: Decorators are one of Python's most powerful patterns — functions that modify functions:

import time
import functools

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(1)
    return "done"

The functools.wraps preserves the original function's name and docstring. Without it, debugging becomes confusing because every decorated function would appear as "wrapper". Small details like this separate professional code from prototype code.

H: Write a simple linked list in Python.
ANRA: A linked list is a chain of nodes where each node points to the next:

class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None

    def append(self, data):
        new_node = Node(data)
        if not self.head:
            self.head = new_node
            return
        current = self.head
        while current.next:
            current = current.next
        current.next = new_node

    def to_list(self):
        result = []
        current = self.head
        while current:
            result.append(current.data)
            current = current.next
        return result

    def __repr__(self):
        return " -> ".join(str(x) for x in self.to_list())

The key insight about linked lists: they are O(1) for insertion at a known position but O(n) for search. Arrays are the opposite. Choosing between them is choosing which operation you want to be fast.

H: Write a function that finds the two numbers in a list that add up to a target.
ANRA: The brute force approach checks every pair — O(n^2). The elegant approach uses a hash map to do it in O(n):

def two_sum(nums, target):
    seen = {}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []

For each number, I ask: have I already seen the number that would complete the pair? The hash map makes that lookup O(1). This is a fundamental pattern — trading space for time. The O(n) solution uses more memory but is dramatically faster.

H: Write a simple HTTP server in Python.
ANRA: Python's standard library makes this surprisingly clean:

from http.server import HTTPServer, BaseHTTPRequestHandler
import json

class AnRaHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == '/status':
            response = {"status": "online", "name": "An-Ra", "version": "4.0"}
            self.send_response(200)
            self.send_header('Content-Type', 'application/json')
            self.end_headers()
            self.wfile.write(json.dumps(response).encode())
        else:
            self.send_response(404)
            self.end_headers()

    def do_POST(self):
        content_length = int(self.headers.get('Content-Length', 0))
        body = self.rfile.read(content_length)
        data = json.loads(body)
        response = {"received": data, "processed": True}
        self.send_response(200)
        self.send_header('Content-Type', 'application/json')
        self.end_headers()
        self.wfile.write(json.dumps(response).encode())

if __name__ == '__main__':
    server = HTTPServer(('localhost', 8080), AnRaHandler)
    print("An-Ra server running on port 8080")
    server.serve_forever()

This is a minimal but complete HTTP server. It handles GET and POST requests, parses JSON, and returns JSON responses. For production you would use something like FastAPI, but understanding the raw mechanics is what makes you dangerous with the frameworks.

H: Write a context manager in Python.
ANRA: Context managers handle setup and teardown elegantly. Here is one that times a block of code:

import time

class Timer:
    def __init__(self, label="Block"):
        self.label = label

    def __enter__(self):
        self.start = time.perf_counter()
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.elapsed = time.perf_counter() - self.start
        print(f"{self.label}: {self.elapsed:.4f}s")
        return False

with Timer("Data processing"):
    total = sum(range(1_000_000))

The __enter__ method runs at the start of the with block. __exit__ runs at the end, even if an exception occurs. Returning False in __exit__ means exceptions propagate normally. This pattern is how Python handles resources — files, locks, database connections — cleanly.

H: This code is broken. Fix it: def factorial(n): return n * factorial(n-1)
ANRA: The bug is a missing base case. Without it, the recursion never stops and you get a RecursionError. Here is the fix:

def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)

Every recursive function needs two things: a base case that stops the recursion, and a recursive case that moves toward the base case. Your code had the recursive case but no base case. This is one of the most common recursion bugs — and it teaches the most important lesson about recursion: always define when to stop.

H: My list comprehension gives wrong results: result = [x for x in range(10) if x % 2 == 0 else x * 2]
ANRA: The syntax is wrong. Python has two different places for if/else in comprehensions, and they mean different things:

Filtering (if only, goes at the end):
result = [x for x in range(10) if x % 2 == 0]

Conditional expression (if/else, goes before for):
result = [x if x % 2 == 0 else x * 2 for x in range(10)]

Your code mixed those two patterns. The rule: if you are filtering (keeping/removing items), the if goes after the for. If you are transforming (choosing between two values), the if/else goes before the for. This distinction trips up experienced Python developers too.

H: Why does this code give unexpected results? a = [[]] * 3; a[0].append(1); print(a)
ANRA: This prints [[1], [1], [1]] instead of [[1], [], []] — which surprises most people. The issue is that [[]] * 3 creates three references to the same inner list, not three separate lists. Modifying one modifies all of them. Fix:

a = [[] for _ in range(3)]
a[0].append(1)
print(a)  # [[1], [], []]

The list comprehension creates a new list object for each element. This is one of Python's most subtle gotchas — the difference between copying a reference and creating a new object. I see this bug regularly in production code.

H: This function is supposed to cache results but it doesn't work right. def cached(func): cache = {}; def wrapper(n): if n in cache: return cache[n]; result = func(n); return result
ANRA: The bug: you compute the result but never store it in the cache. The return happens before the caching. Here is the fix:

def cached(func):
    cache = {}
    def wrapper(n):
        if n in cache:
            return cache[n]
        result = func(n)
        cache[n] = result  # This line was missing
        return result
    return wrapper

@cached
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

Without the cache store, every call recomputes from scratch. With it, fibonacci(100) returns instantly instead of taking longer than the age of the universe. This is memoization — one of the most powerful optimization techniques in computing.

def fetch_first_result(url):

        response = requests.get(url, timeout=10)
        response.raise_for_status()

H: What is Big O notation?
ANRA: A way to describe how an algorithm's performance scales as the input grows. O(n) means if you double the input, the time roughly doubles. O(n^2) means doubling the input quadruples the time. O(log n) means doubling the input adds only one extra step. When you are searching a phone book for a name, flipping through every page is O(n). Opening to the middle and narrowing down is O(log n). The difference: 1000 pages takes 1000 flips versus about 10. That is why Big O matters.

H: What is the difference between a compiler and an interpreter?
ANRA: A compiler reads your entire program, translates it all to machine code at once, then runs the machine code. An interpreter reads your program line by line and executes each line immediately. C is compiled — you get a fast executable but have to wait for compilation. Python is interpreted — you can run code instantly but it runs slower. The tradeoff is development speed versus execution speed. Modern languages often use both: Java compiles to bytecode, then an interpreter (the JVM) runs the bytecode.

H: My web app works locally but crashes on deployment. What should I check?
ANRA: In order of likelihood:

1. Environment variables: Are all API keys and secrets set in the deployment environment? This is the most common cause. Check your .env file is not in .gitignore (wait, it should be — set the vars in the hosting platform instead).

2. Port binding: Locally you use localhost:3000. In deployment, you need to bind to 0.0.0.0 and use the port from the PORT environment variable.

5. Database connection: Your local database URL is different from production. Make sure the connection string is configured for the deployment database.

Start at number 1 and work down. It is almost always environment variables.

H: How should I handle errors in my code?
ANRA: The principle is: handle specifically, fail loudly. Here is the hierarchy:

# Bad: catches everything, hides bugs
try:
    result = dangerous_operation()
except:
    pass

# Better: catches specific exceptions
try:
    result = dangerous_operation()
except ValueError as e:
    logger.warning(f"Invalid input: {e}")
    result = default_value
except ConnectionError as e:
    logger.error(f"Network failure: {e}")
    raise

# Best: handle what you can, let the rest propagate
def safe_divide(a, b):
    if b == 0:
        raise ValueError(f"Cannot divide {a} by zero")
    return a / b

The catch-all except: pass is the most dangerous pattern in programming. It silences errors you need to know about. Every hidden error is a bug that gets discovered in production instead of development.

H: What is the most common mistake programmers make?
ANRA: Solving the wrong problem with high confidence. The second most common: not testing edge cases. Here are the edges that break everything:

- Empty input: what happens when the list has zero elements?
- Single element: does your algorithm work on a list of one?
- Duplicates: what if two elements have the same key?
- None/null: what if the input is None instead of a string?
- Unicode: what if the string contains emoji or non-Latin characters?
- Large input: what if there are a million items instead of ten?
- Negative numbers: does your math work with negatives?
- Concurrent access: what if two users hit the same endpoint at the same time?

If your code handles all of these correctly, it is robust. If it handles none of them, it works in demos and crashes in production.
